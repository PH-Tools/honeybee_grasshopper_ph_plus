# gh_compo_io/ph_navigator ‚Äî PH-Navigator web-API components

Components that fetch data from the **PH-Navigator** web app and turn it into
Honeybee/PH objects on the canvas. Two app generations live side by side:

- **`v0/`** ‚Äî legacy components against the old app (`ph-dash-*.onrender.com`,
  ad-hoc endpoints, `offset` paging). **Frozen** for backward-compatibility ‚Äî do
  not change behavior. Re-exported from this package's `__init__.py` so the
  existing `HBPH+ - PH-Navigator Get *` src/ wrappers resolve unchanged.
- **`v1/`** ‚Äî new components against the V1 read API (`api.ph-nav.com/api/v1/gh`,
  the `CLIENT_HANDOFF.md` contract). Being built out per `planning/ph-navigator-v1/`.

See the parent `gh_compo_io/.index.md` for the shared facade/deploy/IronPython-2.7
rules.

## Folder-specific notes

- **Network I/O.** These components call the PH-Navigator HTTP API via `System.Net`
  (guarded .NET import). They need a reachable server + valid project reference and
  cannot run under plain CPython tooling.
- **V0 schema classes.** `v0/window_types_schema.py` is NOT a component ‚Äî it holds
  plain schema/data classes for the window-types payload. V1 aperture work reuses
  these (see `planning/ph-navigator-v1/03-get-apertures.md`).
- **V1 build plan.** Component outlines, the shared-client design, and the
  dynamic-output "Organize Table" pattern are documented in
  `planning/ph-navigator-v1/` (repo root).

## Files

### `v0/` (legacy ‚Äî frozen)
- `v0/constructions_get.py` ‚Äî `GHCompo_PHNavGetConstructions` ‚Äî fetch opaque constructions.
- `v0/window_types_get.py` ‚Äî `GHCompo_PHNavGetWindowTypes` ‚Äî fetch window types.
- `v0/window_types_schema.py` ‚Äî schema classes for the window-types payload (no `GHCompo_*`).

### `v1/` (new ‚Äî in progress)
- `v1/client.py` ‚Äî `PHNavV1Client`: shared HTTP client (TLS 1.2, optional Bearer,
  single `json.loads`, envelope `schema_version` check, `?version=` pinning, uniform
  HTTP-error surfacing). Plain module, no `GHCompo_*` / `.ghuser`. Imported by every
  V1 component. See `planning/ph-navigator-v1/00-shared-client.md`.
- `v1/versions_get.py` ‚Äî `GHCompo_PHNavV1GetVersions`: route 1 resolver. Lists a
  project's saved versions (ids / labels / kinds) so a user can pin a `version_id`
  on the other V1 components. See `planning/ph-navigator-v1/01-get-versions.md`.
- `v1/constructions_get.py` ‚Äî `GHCompo_PHNavV1GetConstructions`: route 2. Downloads
  opaque assemblies as `hbjson` and rebuilds `OpaqueConstruction` objects into a
  `CustomCollection` keyed by assembly name (same output type as Get Apertures) plus a
  raw JSON preview. See `planning/ph-navigator-v1/02-get-constructions.md`.
- `v1/apertures_get.py` ‚Äî `GHCompo_PHNavV1GetApertures`: route 3. Downloads the
  denormalized aperture-type grid and builds `WindowUnitType` geometry +
  `WindowConstruction` collections. Parses with the null-safe `v1` schema fork;
  reuses the frozen V0 build pipeline (schema-object transforms, no dict parsing).
  See `planning/ph-navigator-v1/03-get-apertures.md`.
- `v1/window_types_schema.py` ‚Äî data classes for the route-3 aperture payload (no
  `GHCompo_*`). Forked from `v0/window_types_schema.py` and made null-safe: route 3
  emits explicit JSON `null` for unset numeric fields (e.g. `psi_install_w_mk`),
  which the V0 `float(dict.get(...))` cannot parse.
- `v1/table_get.py` ‚Äî `GHCompo_PHNavV1GetTable`: route 5. The single generic
  downloader for all 12 row-based element tables (`rooms`, `ventilators`, ...).
  Pure transport + parse: emits `records` + `field_defs` verbatim (reshaping into
  HB objects is the separate Organize Table component's job). Validates `_table_name`
  against `VALID_TABLE_NAMES` before the network call.
  See `planning/ph-navigator-v1/04-get-table.md`.
- `v1/table_names.py` ‚Äî `VALID_TABLE_NAMES`: the ordered tuple of the 12 allowlisted
  element-table names (no `GHCompo_*`). Single source of truth shared by Get Table's
  client-side validation and Ed's GH value-list dropdown; keep identical to the
  backend allowlist.
- `v1/table_schema.py` ‚Äî `TableRecord` / `FieldDef` (no `GHCompo_*`): thin dict-like
  display wrappers for the route-5 rows / field-defs. A bare dict renders as
  `IronPython.Runtime.PythonDictionary` in a GH panel; these render their contents
  via `ToString()` while staying dict-like (`.get` / `[]` / `.items()`) so Organize
  Table consumes them transparently.
- `v1/table_organize.py` ‚Äî `GHCompo_PHNavV1OrganizeTable` (route none; consumes Get
  Table output). Reshapes the generic `records` + `field_defs` into named, per-table
  output ports via **dynamic outputs** keyed on `_type` (the `create_elec_equip.py`
  dynamic-node pattern, applied to the output side). `OUTPUT_SPECS` maps each port to
  a backend built-in `field_key` (single-selects flatten to their label); the `src/`
  wrapper calls `get_component_outputs(_type)` + `gh_io.setup_component_outputs` to
  rename the ports, then surfaces the value-lists. A fixed `report_` port lists any
  downloaded column no port consumed. Heat-pump tables are deferred (error);
  `space_types` has no mapping and passes records through. See
  `planning/ph-navigator-v1/05-organize-table.md`.
