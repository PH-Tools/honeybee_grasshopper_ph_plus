# gh_compo_io — component logic backend (the "why" of the two-layer design)

This folder holds the **actual logic** for every HBPH+ Grasshopper component. Each
file exports one `GHCompo_*` class that does the real work. The `.ghuser`
components a user drops on the Grasshopper canvas are just **thin facades** that
import and call these classes.

If you are adding or editing a component, read this file first — the structure
here is deliberate and a little unusual, and the reasons matter.

---

## Why two layers? (facade on canvas, logic in this repo)

A Grasshopper component's Python lives *inside* the `.ghuser` binary, which lives
*inside* Rhino's application-support folders. That is a terrible place to keep
source: it can't be diffed, can't be code-reviewed, can't be version-controlled,
and can't be edited in a real IDE with type-checking.

So we split every component in two:

1. **The canvas facade** — `honeybee_grasshopper_ph_plus/src/HBPH+ - <Name>.py`.
   A thin, mostly-boilerplate script compiled into the `.ghuser`. It wires up the
   Grasshopper environment, instantiates a `GHCompo_*` class from *this* folder,
   passes the canvas inputs straight through, and returns the result. **No real
   logic lives here.**

2. **The logic class** — `gh_compo_io/<subcat>/<name>.py` (this folder).
   A normal, importable, type-checkable Python module holding the `GHCompo_*`
   class. This is where all the work happens, and it is what we actually maintain,
   review, and test.

The facade is dumb on purpose. It changes almost never. The logic class is where
you do 99% of your editing, and because it is an ordinary module in a git repo,
you get diffs, review, IDE autocomplete, and static analysis.

## How the code gets *into* Rhino: fsdeploy

This repo is **never** edited from inside the Rhino folders. We edit here, in git,
and a VS Code extension (**fsdeploy**, configured in `.vscode/settings.json`)
copies the package to Rhino's runtime paths on every save (`deployOnSave: true`).

Current deploy targets (see `.vscode/settings.json` → `fsdeploy.nodes`):

- `…/ladybug_tools/python/.../site-packages/` — the Rhino/Ladybug-Tools Python
  path. This is the copy Grasshopper actually imports at runtime.
- `…/PHX/.venv/.../site-packages/` — the sibling PHX repo's venv, so PHX-side dev
  and tests resolve the same package.

Consequence: **saving a file here silently updates the live Rhino install.** There
is no build step for logic changes — edit the `GHCompo_*` class, save, and the
next component solve in Grasshopper picks it up. (The `.ghuser` facade is the only
thing that needs a real rebuild, and only when you change the facade itself — see
below.)

## Runtime target: IronPython 2.7 (write Py2-compatible code)

The deployed code is imported by Grasshopper's **IronPython 2.7** interpreter, not
CPython 3. Everything in this folder must be Py2.7-safe even though our `.venv`,
linters, and type-checker are CPython. Concretely, across these files you will
see — and must keep using — these idioms:

- **No f-strings.** Use `"{}".format(...)`.
- **Type hints in `# type:` comments**, never inline annotations
  (`def run(self):  # type: () -> int | float | None`). The `typing` import itself
  is wrapped in `try/except ImportError: pass  # IronPython 2.7`, because IPy2.7
  has no `typing` at runtime — it only needs to resolve for the CPython checker.
- **Guarded optional imports.** `.NET`/Rhino imports
  (`System.Net`, `Grasshopper`, `Rhino`) are wrapped in
  `try/except: pass  # Outside Rhino` so the module still imports under plain
  CPython for tooling. Cross-package imports (`honeybee_ph_rhino`, `ph_units`, …)
  are wrapped in `try/except ImportError: raise ImportError("Failed to import …")`
  to give a legible error on the canvas when a sibling package is missing.
- **Py2 stdlib fallbacks**, e.g. `from itertools import izip`/`izip_longest` with a
  `zip`/`zip_longest` fallback.

Do **not** "modernize" any of this. Match the surrounding style exactly.

## Anatomy of a `GHCompo_*` logic class

The convention is uniform enough that you can copy an existing neighbor. Using
`hb_tools/convert_unit.py` as the reference shape:

```python
class GHCompo_ConvertValueToUnit(object):
    def __init__(self, _IGH, _in, *args, **kwargs):
        # type: (gh_io.IGH, str, *Any, **Any) -> None
        self.IGH = _IGH          # the Grasshopper interface object (see below)
        self._in = _in           # canvas inputs, passed straight through

    @property
    def ready(self):             # optional guard — bail cleanly on missing inputs
        # type: () -> bool
        return self._in is not None

    def run(self):               # THE entry point the facade calls
        # type: () -> ...
        ...
        self.IGH.remark(msg)     # user-facing messages go through IGH, not print
        return result
```

Conventions worth preserving:

- **Class name** is `GHCompo_<PascalCaseName>`. It is exported from the
  subfolder's `__init__.py` (which the top-level `gh_compo_io/__init__.py` re-exports
  with `*`), so the facade can reach it as `gh_compo_io.GHCompo_<Name>`.
- **Constructor signature mirrors the canvas inputs**, in order, plus the leading
  `_IGH`. Trailing `*args, **kwargs` are common so the facade can pass extra
  inputs without breaking older signatures.
- **`_IGH` is the one hard dependency.** It's an `honeybee_ph_rhino.gh_io.IGH`
  instance built by the facade — the sanctioned gateway to the Rhino/GH API
  (geometry coercion, `remark`/`warning` bubbles, the active `ghdoc`, etc.). Use
  it instead of importing `Rhino`/`Grasshopper` directly, and use `IGH.remark(...)`
  instead of `print` for anything the user should see.
- **`run()` returns the output value(s)** the facade assigns to its `out_` (or
  named) canvas outputs. Return `None` (not an exception) for the ordinary
  "inputs not ready yet" case.

## What the facade does (so you know what you *don't* write here)

The `src/HBPH+ - <Name>.py` facade is fixed boilerplate:

1. GPL-3.0 header + a docstring whose `Args:`/`Returns:` block **Grasshopper parses
   into the canvas tooltips** — so that docstring is real UI, not decoration.
2. Sets `ghenv.Component.Name = "HBPH+ - <Name>"` and calls
   `_component_info_.set_component_params(ghenv, dev=False)`, which stamps the
   NickName / Category / SubCategory / version message from the registry in
   `_component_info_.py`. The `Name` string must match the registry key **and** the
   filename exactly.
3. Builds `IGH = gh_io.IGH(ghdoc, ghenv, sc, rh, rs, ghc, gh)`.
4. Instantiates `gh_compo_io.GHCompo_<Name>(IGH, <inputs…>)` and calls `.run()`.

The `if DEV:` block (guarded by `dev=False`) is a **hot-reload aid for canvas
development only**: it `reload()`s the logic module so edits show up without
restarting Rhino. Leave `dev=False` on commit.

## Adding a new component — the full checklist

Editing behavior = edit the `GHCompo_*` class here and save (fsdeploy handles the
rest). **Adding** a component touches more files (see the repo-root `.index.md` and
`CLAUDE.md`):

1. New logic class → `gh_compo_io/<subcat>/<name>.py` (this folder).
2. Export it from that subfolder's `__init__.py`.
3. New facade → `honeybee_grasshopper_ph_plus/src/HBPH+ - <Name>.py` (copy a
   neighbor; change the Name, docstring, class, and inputs).
4. Register it in `honeybee_ph_plus_rhino/_component_info_.py` (`COMPONENT_PARAMS`
   entry: NickName / Category / SubCategory).
5. Add an icon (`honeybee_grasshopper_ph_plus/icons/`).
6. Rebuild the `.ghuser` (compiled from the facade — the only real build step).

## Subcategory folders

Each subfolder groups components by canvas subcategory and has its own `.index.md`
with folder-specific notes:

- `airtable/` — pull records from AirTable bases into HB/PH objects (network I/O).
- `collections/` — generic keyed-collection container components (pure Python).
- `ghpy/` — read/write arbitrary attributes on raw Python objects on the canvas.
- `hb_tools/` — the grab-bag: geometry cleanup, window builders, unit conversion,
  E+ SQL readers, sorting/grouping helpers (the largest folder).
- `ph_navigator/` — fetch constructions/window-types from the PH-Navigator web API.
- `read/` — read PHPP data via an out-of-process shell/subprocess bridge.
- `reporting/` — build PDF report geometry, annotations, and Rhino layout output.

`run_subprocess.py` (folder root) is a shared helper for the `read/` components
that shell out to an external process.
