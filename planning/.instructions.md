# planning/ — working plans and in-flight work

Tracked planning material for HBPH+. Distinct from `context/` (stable canon).

## Source-of-truth rules

1. **Plans are provisional; `context/` is canonical.** When a plan lands, fold the outcome into the relevant `context/` doc.
2. **`STATUS.md` is the index.** Every active item gets a row.
3. **One folder per unit of work**, lowercase kebab-case.

## Layout

```
planning/
  README.md          # read order
  STATUS.md          # master index of active work
  features/          # new-capability plans   (features/<slug>/)
  archive/           # completed/superseded    (archive/<slug>/ + README index)
```

## Feature folder contract

```
<slug>/
  README.md   # router: scope, read order
  PRD.md      # what/why
  STATUS.md   # current state, next step, blockers
  PLAN.md     # optional implementation sequence
```

## Archiving completed work

When an item is `Complete`/`Superseded`: fold the outcome into `context/`, then **move** its folder to `planning/archive/<slug>/` unchanged and add a row to `planning/archive/README.md`. Flat by slug, never nested by date. (The existing `archive/ph-navigator-v1/` follows this shape.)

## Status vocabulary

`Requested` · `Scoped` · `In progress` · `Blocked` · `Implemented` · `Complete` · `Deferred` · `Superseded`.
